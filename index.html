📖 Les fondamentaux de Kubernetes

            

                Kubernetes (souvent abrégé K8s) est un orchestrateur de conteneurs open-source.
                Il automatise le déploiement, la mise à l’échelle et la gestion des applications conteneurisées.
                Cette section vous propose des fiches détaillées sur chaque concept et chaque commande.
            


            
            
🧩 Concepts clés

            

                

                    
Concept Pod

                    
Le Pod est l’unité la plus petite déployable. Il contient un ou plusieurs conteneurs (ex: Docker) qui partagent le même réseau et le même stockage. Un Pod représente un processus applicatif.

                    
Exemple commenté :

                    
apiVersion: v1              # Version de l'API Kubernetes
kind: Pod                   # Type de ressource : Pod
metadata:
  name: mon-pod             # Nom unique du Pod
  labels:
    app: nginx              # Label pour identifier le Pod
spec:
  containers:
  - name: nginx             # Nom du conteneur
    image: nginx:latest     # Image Docker
    ports:
    - containerPort: 80     # Port exposé par le conteneur

                

                

                    
Concept Service

                    
Un Service expose un ensemble de Pods via une adresse IP stable et un nom DNS. Types : ClusterIP (interne), NodePort (externe via port), LoadBalancer (avec IP externe).

                    
Exemple commenté :

                    
apiVersion: v1
kind: Service
metadata:
  name: mon-service
spec:
  selector:
    app: nginx              # Sélectionne les Pods avec ce label
  ports:
  - protocol: TCP
    port: 80                # Port exposé par le service
    targetPort: 80          # Port du conteneur
  type: ClusterIP           # Par défaut, interne au cluster

                

                

                    
Concept Ingress

                    
Un Ingress expose des services HTTP/HTTPS à l’extérieur. Il agit comme un routeur (reverse proxy) qui redirige le trafic en fonction du nom d’hôte ou du chemin.

                    
Exemple commenté :

                    
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: mon-ingress
spec:
  rules:
  - host: monapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: mon-service
            port:
              number: 80

                

                

                    
Concept Volume & StorageClass

                    
Les Volumes persistent les données au‑delà du cycle de vie d’un Pod. Une StorageClass définit un provisionneur pour créer automatiquement des volumes.

                    
Exemple de StorageClass :

                    
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: local-path
provisioner: rancher.io/local-path
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer

                

                

                    
Concept ConfigMap & Secret

                    
Les ConfigMap stockent des données de configuration non sensibles. Les Secret stockent des données sensibles (mots de passe, tokens) encodées en base64.

                    
Exemple de ConfigMap :

                    
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  APP_COLOR: blue
  APP_MODE: production

                

                

                    
Concept Deployment

                    
Un Deployment gère un ensemble de Pods (ReplicaSet) avec des fonctionnalités de mise à jour rolling, rollback, et scaling.

                    
Exemple commenté :

                    
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3               # Nombre de réplicas
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80

                

            


            
            
🛠️ Commandes kubectl – explications et exemples

            
Voici les commandes essentielles, avec des commentaires pédagogiques et des exemples concrets.


            
1. Vérifier l'état du cluster

            

                
🔍 kubectl cluster-info

                
Explication : Affiche les adresses des composants du plan de contrôle (API Server, etcd, Scheduler, etc.).

                
Exemple :

                
$ kubectl cluster-info
Kubernetes control plane is running at https://172.18.0.2:6443
CoreDNS is running at https://172.18.0.2:6443/api/v1/namespaces/kube-system/services/kube-dns:dns/proxy

                
💡 Commentaire : Cette commande confirme que le cluster est accessible et vous donne l'URL du serveur API.

            


            
2. Lister les ressources

            

                
📋 kubectl get <ressource>

                
Explication : Affiche la liste des ressources d'un type donné (nodes, pods, services, deployments, etc.).

                
Exemples :

                
$ kubectl get nodes
NAME       STATUS   ROLES                  AGE   VERSION
kind-control-plane   Ready    control-plane   10m   v1.27.3
kind-worker          Ready    <none>         10m   v1.27.3

$ kubectl get pods -n kube-system
NAME                             READY   STATUS    RESTARTS   AGE
coredns-...                      1/1     Running   0          10m
...

                
💡 Commentaire : L'option -n permet de spécifier un namespace. -o wide donne plus de détails (ex: IP, nœud).

            


            
3. Obtenir des détails sur une ressource

            

                
🔎 kubectl describe <ressource> <nom>

                
Explication : Fournit des informations détaillées (statut, événements, conditions) sur une ressource spécifique.

                
Exemple :

                
$ kubectl describe pod mon-pod
Name:         mon-pod
Namespace:    default
Node:         kind-worker/...
...
Events:
  Type    Reason     Age   From               Message
  ----    ------     ----  ----               -------
  Normal  Scheduled  2m    default-scheduler  Successfully assigned default/mon-pod to kind-worker
  Normal  Pulling    2m    kubelet            Pulling image "nginx:latest"
  Normal  Started    1m    kubelet            Started container nginx

                
💡 Commentaire : La section Events est précieuse pour diagnostiquer les problèmes (ex: image introuvable, nœud non prêt).

            


            
4. Appliquer un fichier de configuration

            

                
📄 kubectl apply -f <fichier>

                
Explication : Crée ou met à jour des ressources à partir d'un fichier YAML ou JSON. C'est la méthode déclarative recommandée.

                
Exemple :

                
$ kubectl apply -f deployment.yaml
deployment.apps/nginx-deployment created

$ kubectl apply -f ingress.yaml
ingress.networking.k8s.io/mon-ingress created

                
💡 Commentaire : Kubernetes garde en mémoire la dernière configuration appliquée, ce qui permet des mises à jour incrémentales.

            


            
5. Supprimer une ressource

            

                
🗑️ kubectl delete <ressource> <nom>

                
Explication : Supprime une ressource. Peut aussi supprimer via fichier : kubectl delete -f fichier.yaml.

                
Exemple :

                
$ kubectl delete pod mon-pod
pod "mon-pod" deleted

$ kubectl delete -f deployment.yaml
deployment.apps "nginx-deployment" deleted

                
💡 Commentaire : La suppression est définitive. Soyez prudent avec les données persistantes (PVC).

            


            
6. Consulter les logs d'un Pod

            

                
📜 kubectl logs <nom-pod>

                
Explication : Affiche les logs du conteneur principal du Pod. Pour un conteneur spécifique : -c <conteneur>.

                
Exemple :

                
$ kubectl logs nginx-deployment-7d8f8-5xg7f
127.0.0.1 - - [10/Jan/2024:12:00:00 +0000] "GET / HTTP/1.1" 200 612

                
💡 Commentaire : Utile pour déboguer les applications. Avec --tail=50 on peut limiter le nombre de lignes.

            


            
7. Accéder à un shell dans un conteneur

            

                
🖥️ kubectl exec -it <nom-pod> -- /bin/bash

                
Explication : Ouvre un terminal interactif à l'intérieur du conteneur, pour exécuter des commandes de diagnostic.

                
Exemple :

                
$ kubectl exec -it mon-pod -- /bin/bash
root@mon-pod:/# ls
bin  dev  etc  home  ...

                
💡 Commentaire : Très utile pour inspecter l'environnement, vérifier des fichiers de configuration, ou tester des connexions réseau.

            


            
8. Exposer un service en local (port-forward)

            

                
🔁 kubectl port-forward <nom-pod> <port-local>:<port-pod>

                
Explication : Redirige un port de votre machine vers un port du Pod, pratique pour tester sans exposet de service externe.

                
Exemple :

                
$ kubectl port-forward mon-pod 8080:80
Forwarding from 127.0.0.1:8080 -> 80
# Dans un autre terminal :
$ curl http://localhost:8080
... (réponse de Nginx)

                
💡 Commentaire : Idéal pour le développement ou les tests ponctuels.

            


            
            
📦 Exemples complets de déploiement


            
Déployer une application Nginx avec un service LoadBalancer

            

                
Étape 1 : Créer le fichier nginx-deploy.yaml

                
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deploy
spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx
  ports:
  - protocol: TCP
    port: 80
    targetPort: 80
  type: LoadBalancer   # Si MetalLB ou cloud provider est installé

                
💡 Ce fichier contient deux ressources : un Deployment et un Service. Le séparateur --- permet de les regrouper.

                
Étape 2 : Appliquer

                
$ kubectl apply -f nginx-deploy.yaml
deployment.apps/nginx-deploy created
service/nginx-service created

                
Étape 3 : Vérifier

                
$ kubectl get pods
NAME                            READY   STATUS    RESTARTS   AGE
nginx-deploy-7d8f8-5xg7f        1/1     Running   0          20s
nginx-deploy-7d8f8-6h8g9        1/1     Running   0          20s

$ kubectl get svc
NAME            TYPE           CLUSTER-IP      EXTERNAL-IP     PORT(S)        AGE
nginx-service   LoadBalancer   10.96.123.45    192.168.1.240   80:30001/TCP   20s

                
💡 L'EXTERNAL-IP est attribuée par MetalLB (ou le cloud provider). Vous pouvez maintenant accéder à l'application via http://192.168.1.240.

            


            
Utiliser un Ingress pour router vers plusieurs services

            

                
Fichier ingress-routes.yaml

                
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: mon-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  rules:
  - host: app1.mondomaine.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: service-app1
            port:
              number: 80
  - host: app2.mondomaine.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: service-app2
            port:
              number: 80

                
💡 Cet Ingress redirige le trafic selon le nom d'hôte. Les services service-app1 et service-app2 doivent exister.

                
Application :

                
$ kubectl apply -f ingress-routes.yaml
ingress.networking.k8s.io/mon-ingress created

$ kubectl get ingress
NAME          CLASS    HOSTS                       ADDRESS         PORTS
mon-ingress   <none>   app1.mondomaine.com,...    192.168.1.240   80

                
💡 L'ADDRESS correspond à l'IP du LoadBalancer de l'Ingress Controller (ici Nginx).

            


            
Utiliser un ConfigMap pour paramétrer un Pod

            

                
ConfigMap (app-config.yaml) :

                
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  APP_ENV: production
  APP_DEBUG: "false"
  DATABASE_URL: postgres://user:pass@db:5432/mydb

                
Pod utilisant ce ConfigMap (pod-with-config.yaml) :

                
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: monapp:latest
    envFrom:
    - configMapRef:
        name: app-config
    # Les variables d'environnement APP_ENV, APP_DEBUG, DATABASE_URL seront injectées

                
💡 Les ConfigMaps découplent la configuration du code. On peut les modifier sans rebuild l'image.

                
Application :

                
$ kubectl apply -f app-config.yaml
configmap/app-config created

$ kubectl apply -f pod-with-config.yaml
pod/app-pod created

$ kubectl exec app-pod -- env | grep APP_
APP_ENV=production
APP_DEBUG=false
DATABASE_URL=postgres://user:pass@db:5432/mydb

            


            
            
📐 Bonnes pratiques

            
    
                
  • Utilisez des labels et annotations pour organiser vos ressources (ex: version, environnement, équipe).
    • 
                
  • Définissez des requests/limits (CPU, mémoire) pour chaque conteneur afin d'éviter qu'un Pod ne consomme toutes les ressources.
    • 
                
  • Préférez les Deployments aux Pods seuls, car ils offrent le rolling update et le rollback.
    • 
                
  • Utilisez des namespaces pour isoler les environnements (dev, test, prod).
    • 
                
  • Activez les Network Policies pour restreindre les communications entre Pods (sécurité).
    • 
                
  • Surveillez votre cluster avec Prometheus/Grafana (comme dans le générateur).
    • 
                
  • Stockez les secrets en base64 (pas en clair) et utilisez des outils comme kubectl create secret.
